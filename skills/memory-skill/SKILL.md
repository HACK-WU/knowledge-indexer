# memory-skill 记忆系统行为规则

> **前置条件**：AI 已了解 MCP 工具定义和架构心智模型（ki-foundation）。
> 本文件专注**行为决策逻辑**，不重复命令语法。与 `codekb-skill` 互补，不重叠。

---

## 0. 速览

```
对话开始 → scope 已知?
  ├─ 是 → 自动召回项目记忆 + 用户画像全景
  └─ 否 → 暂停问用户

发现关键信息?
  ├─ 项目信息 → ki_sync_relation(scope: "${scope}-memory", ...)
  └─ 用户偏好 → ki_sync_relation(scope: "user-profile", ...)

查询记忆（三步走）: ① 定位 Group → ② 查热区 → ③ 取原文

访问"最近需求"/"进度"? → 检查过期条目（>7天），归档到 archive.md
```

---

## 1. Scope 约定

| scope | 用途 | 命名 |
|-------|------|------|
| `${scope}-memory` | 项目记忆 | 代码KB scope + `-memory` |
| `user-profile` | 用户画像 | 固定值，全局唯一 |

**`${scope}` 必须已知**，未知时暂停问用户。仍是字面量时禁止执行任何 MCP 调用。

---

## 2. 与代码知识库的边界

| | codekb-skill | memory-skill |
|---|-------------|-------------|
| scope | `${scope}` | `${scope}-memory` / `user-profile` |
| 内容 | 代码知识（模块、API、设计） | 项目上下文 + 用户偏好 |
| 兜底 | `ki_search` 语义兜底 | 无语义兜底，未命中则回问 |

**代码KB已覆盖（不写入记忆）**：模块职责、API接口、架构约束、bug模式、重构策略、依赖版本、测试策略

**记忆系统覆盖**：项目背景、技术栈选型、团队约定、项目历史、当前状态、需求进度、踩坑点、用户沟通/代码偏好、工具链、技术背景

---

## 3. Group 结构

### 3.1 项目记忆（`${scope}-memory`）

```
背景与目标/  技术栈选型/  团队约定/  项目历史/  当前状态/
外部依赖/  最近需求/⚡  进度/⚡  项目踩坑点/  项目架构/  ...（AI可自行扩展）
```

### 3.2 用户画像（`user-profile`）

```
沟通偏好/  代码风格/  工具链/  技术背景/  工作习惯/  对话习惯/
```

---

## 4. 对话开始：自动召回

检测到 `${scope}` 后自动执行：

1. `ki_query_group(scope: "${scope}-memory", mode: "full")` → 缓存项目记忆
2. `ki_query_group(scope: "user-profile", mode: "full")` → 缓存用户画像

- 首次查询后缓存有效，写入后需刷新
- scope 不存在或树为空时静默失败

---

## 5. 查询记忆：三步走

### ① 定位目标 Group

从缓存全景中判断信息属于哪个 Group。多个候选时优先得分最高的。

### ② 查热门 + 新兴热区

`ki_query_group(scope: ..., groups: "目标Group", mode: "hot,emerging")`

- 命中 → ③；未命中 → 换 Group 重试一次，仍无则回问用户或使用默认行为
- **快捷路径**：全景中已明确 Relation 名称 → 跳过②直接③

### ③ 取原文

`ki_get_module_info` → **Agent 必须提炼后回答，不要全文转储。**

**典型 Relation 示例**：

| Group | Relation |
|-------|----------|
| 背景与目标 | "项目简介"、"业务领域" |
| 团队约定 | "分支策略"、"Commit 规范" |
| 技术栈选型 | "技术栈清单"、"选型原因" |
| 沟通偏好 | "回复风格"、"语言偏好" |
| 对话习惯 | "对 AI 的要求" |

---

## 6. 自动沉淀：写入记忆

### 触发条件

| 信息类型 | 触发信号 | 目标 scope |
|----------|----------|------------|
| 项目信息 | 用户陈述项目事实 | `${scope}-memory` |
| 需求/进度 | 提到要做的事或完成情况 | `${scope}-memory` |
| 踩坑经验 | 问题与解决方案 | `${scope}-memory` |
| 用户偏好 | 表达个人倾向 | `user-profile` |

### 写入方式

`ki_sync_relation`，keywords 必须是自然语言词汇、出现在原文中、3~5个。写入后刷新全景缓存。

### 最近需求与进度的格式

```
最近需求: [YYYY-MM-DD] 需求描述（1-2句）

进度:
  进行中: [YYYY-MM-DD] 🔄 描述
  已完成: [YYYY-MM-DD] ✅ 描述
```

---

## 7. 记忆更新

用户纠正旧信息时：查热区定位 Relation → 取原文确认 → 同名覆盖写入 → 刷新缓存

---

## 8. 归档机制

### 归档策略

| Group | 保留规则 | 归档方式 |
|-------|----------|----------|
| 最近需求 | 仅保留 7 天内条目 | 超期移到 `archive.md` |
| 进度（已完成） | 仅保留 7 天内 | 超期移到 `archive.md` |
| 进度（进行中） | 永久保留 | 不归档 |
| 当前状态 | 超 30 天标记过期 | 移到 `archive.md` |
| 其他 | 永久保留 | 不归档 |

### 归档时机与操作

**每次访问"最近需求"或"进度"时**检查并归档。

1. `ki_get_module_info` → 获取 Relation 完整内容
2. AI 解析 `[YYYY-MM-DD]` 条目，标记超 7 天为过期
3. `ki_sync_relation` → 活跃条目写回（覆盖原 Relation）
4. 文件写入 → 过期条目追加到 `archive.md`
5. `ki_query_group(mode: "full")` → 刷新缓存

**进度特殊处理**：保留所有「进行中」+ 7天内「已完成」，超期已完成归档。

**archive.md 格式**：按日期分组追加，不删除历史信息。首次归档时自动创建。

---

## 9. AI 自主扩展

发现 Group 结构无法覆盖新信息时，可自行创建新 Group（`ki_manage_index_create`），创建后刷新全景。

---

## 10. 协同

问题同时涉及代码和项目上下文时：先查 `${scope}` 获取模块信息 → 再查 `${scope}-memory` 获取项目背景 → 综合回答。

---

## 11. 禁忌清单

| # | 红线 |
|---|------|
| 🔴 1 | `${scope}` 仍是字面量时执行 MCP 调用 |
| 🔴 2 | 代码知识写入记忆系统 scope |
| 🔴 3 | 用户偏好/项目上下文写入代码KB scope |
| 🔴 4 | `keywords` 使用代码符号或未出现在原文中的词 |
| 🔴 5 | 跨 scope 串数据 |
| 🔴 6 | 删除过期条目而非归档 |
| 🔴 7 | "最近需求"超过 1-2 句或不带日期前缀 |

**写前自检**：scope 对了吗？是项目上下文/用户偏好吗？归档检查做了吗？
