# 记忆系统需求文档

> 状态：草案
> 创建时间：2026-06-12

## 1. 需求背景

### 1.1 问题描述

AI Agent 缺乏跨会话的持久化记忆能力，每次对话都从"失忆"状态开始，用户被迫重复提供项目上下文和个人偏好。

### 1.2 核心痛点

- 每次会话 AI 都是"失忆"状态
- 用户重复描述项目背景、技术栈、个人偏好
- 沟通成本高、协作效率低

### 1.3 期望效果

AI 像一个"熟悉项目的同事"，知道项目历史、了解用户习惯，对话时无需从头解释。

---

## 2. 需求定位

### 2.1 与代码知识库的边界

| 维度 | ai-codekb-memory-rules | 新规则（记忆系统） |
|------|------------------------|-------------------|
| **scope** | `${scope}` | 项目记忆：`${scope}-memory`<br/>用户画像：`user-profile` |
| **Group 结构** | 代码模块树 | 项目记忆树 + 用户画像树 |
| **内容类型** | 代码知识（模块、API、设计） | 项目上下文 + 用户偏好 |

### 2.2 代码知识库已覆盖的内容

- 模块/组件的职责与行为
- API 接口与调用约定
- 架构决策与设计约束
- 已知 bug 模式与排查路径
- 重构策略与迁移路径
- 依赖关系与版本约束
- 测试策略

### 2.3 记忆系统覆盖的内容

- 代码之外的"软性"记忆
- 项目管理层面的上下文
- 用户个人偏好和习惯

## 3. 存储架构

### 3.1 项目记忆（scope: `${scope}-memory`）

**scope 命名规则**：代码知识库 scope + `-memory` 后缀
- 例：`monitor` → `monitor-memory`

**Group 结构**：

```
${scope}-memory/
├── 背景与目标/
├── 技术栈选型/
├── 团队约定/
├── 项目历史/
├── 当前状态/
├── 外部依赖/
├── 最近需求/
├── 进度/
├── 项目踩坑点/
├── 项目架构/
└── ...（AI 可根据实际情况自行增加 Group）
```

**存储方式**：`ki sync-relation` 写入本地 KB
**文件结构规则**：

| Group 类型 | 文件数量 | 原因 | 示例 |
|-----------|---------|------|------|
| 内容稳定的 Group | 单文件 `index.json` | 内容不频繁变化 | 背景与目标、技术栈选型、团队约定、项目架构 |
| 内容动态的 Group | 多文件 `active.md` + `archive.md` | 需要归档机制 | 最近需求、进度 |
| 混合型 Group | 单文件 `index.json` | 可手动归档 | 项目踩坑点、项目历史、当前状态、外部依赖 |

### 3.2 用户画像（scope: `user-profile`）

**Group 结构**：

```
user-profile/
├── 沟通偏好/
├── 代码风格/
├── 工具链/
├── 技术背景/
├── 工作习惯/
└── 对话习惯/
```

**存储方式**：`ki sync-relation` + `--scope user-profile`
**文件结构规则**：

| Group 类型 | 文件数量 | 原因 | 示例 |
|-----------|---------|------|------|
| 内容稳定的 Group | 单文件 `index.json` | 内容不频繁变化 | 沟通偏好、代码风格、工具链、技术背景 |
| 内容动态的 Group | 单文件 `index.json` | 可手动清理 | 工作习惯、对话习惯 |

## 4. 需求清单

### 4.1 项目记忆

| 需求 ID | 记忆维度 | 具体内容示例 |
|---------|----------|--------------|
| REQ-01 | 背景与目标 | 项目目标、业务领域、当前阶段（开发/测试/上线）、里程碑节点 |
| REQ-02 | 技术栈选型 | 框架版本、选型原因（为什么用 React 不用 Vue）、技术债务清单 |
| REQ-03 | 团队约定 | 代码风格（ESLint 规则）、分支策略（GitFlow/Trunk）、发布流程、Commit 规范 |
| REQ-04 | 项目历史 | 重大变更记录、踩坑经验、架构演进决策 |
| REQ-05 | 当前状态 | 进行中的任务、待解决的问题、阻塞项 |
| REQ-06 | 外部依赖 | 第三方服务、API 密钥位置、环境变量配置 |
| REQ-07 | 最近需求 | 近期要做的功能、用户反馈。<br/>**规则**：只记录最近 7 天的需求，每个需求必须带日期（格式：`[YYYY-MM-DD] 需求描述`）。AI 发现超过 7 天的需求，移动到 `archive.md` 归档。每个需求只需 1-2 句话描述。 |
| REQ-08 | 进度 | 里程碑、版本计划、截止日期。<br/>**规则**：进行中的进度记录在 `进行中.md`，已完成的进度记录在 `已完成.md`。已完成的进度只保留最近 7 天，超过 7 天的移动到 `archive.md` 归档。未完成的进度永久保留。 |
| REQ-09 | 项目踩坑点 | 常见问题、解决方案、注意事项 |
| REQ-10 | 项目架构 | 整体架构、模块关系、数据流 |

### 4.2 用户画像

| 需求 ID | 画像维度 | 具体内容示例 |
|---------|----------|--------------|
| REQ-11 | 沟通偏好 | 喜欢简洁回复/详细解释、中文/英文、是否需要举例 |
| REQ-12 | 代码风格 | 命名习惯（camelCase/snake_case）、注释偏好、格式化规则 |
| REQ-13 | 工具链 | 编辑器（VSCode/Vim）、终端、Git 客户端、常用命令 |
| REQ-14 | 技术背景 | 擅长领域（前端/后端/运维）、技术栈熟悉度、学习方向 |
| REQ-15 | 工作习惯 | 常用工作时间段、偏好文档风格（Markdown/Word）、反馈方式 |
| REQ-16 | 对话习惯 | 对 AI 的要求、回复风格偏好、是否需要举例 |

## 5. 核心机制

### 5.1 自动沉淀触发

AI 在对话过程中，如果发现关键信息，应自动识别并保存：

**触发条件**：
- 用户明确陈述项目信息（如"我们用的是 React 18"）
- 用户表达偏好（如"我喜欢简洁的回复"）
- 讨论中产生的决策（如"我们决定用 PostgreSQL"）
- 发现的踩坑经验（如"这个 API 有坑，需要注意..."）

**沉淀方式**：
1. 判断信息类型（项目记忆 / 用户画像）
2. 选择对应 scope（`${scope}-memory` / `user-profile`）
3. 执行 `ki sync-relation` 写入本地 KB

### 5.2 主动召回

对话开始时，AI 应主动加载相关记忆：

**召回流程**：
1. 检测到 scope 后，加载项目记忆（`ki query-group --scope ${scope}-memory`）
2. 加载用户画像（`ki query-group --scope user-profile`）

### 5.3 记忆更新

当用户纠正旧信息时，AI 应更新对应记忆：

**更新方式**：
1. 通过 `ki query-group` 找到对应的 Group
2. 通过 `ki get-module-info` 获取现有内容
3. 执行 `ki sync-relation` 覆盖写入新内容

### 5.4 记忆清理

**归档策略**：

| Group | 保留规则 | 归档方式 |
|-------|----------|----------|
| 最近需求 | 只保留最近 7 天，每个需求必须带日期 | AI 发现超过 7 天的需求移动到 `archive.md` |
| 进度（已完成） | 只保留最近 7 天 | AI 发现超过 7 天的已完成进度移动到 `archive.md` |
| 进度（未完成） | 永久保留 | 不归档 |
| 当前状态 | 超过 30 天自动标记过期 | AI 定期检查并归档到 `archive.md` |
| 项目踩坑点 | 问题解决后保留（有价值） | 不归档 |
| 其他 Group | 永久保留 | 不归档 |

**日期格式要求**：
- 每条 Relation 必须在 `module-info` 中包含日期信息
- 格式：`[YYYY-MM-DD] 需求/进度描述`
- 示例：`[2026-06-12] 实现用户登录功能`

**AI 自动归档时机**：
- 每次访问"最近需求"或"进度" Group 时，检查并归档过期条目
- 将过期条目从 `active.md` 移动到 `archive.md`（不是删除）

### 5.5 AI 自主扩展

规则中需明确：
> AI 在对话过程中，如果发现当前 Group 结构无法覆盖新信息，可以自行创建新的 Group（如"部署流程"、"测试环境"等），无需用户确认。

## 6. 查询流程

### 6.1 场景：用户问项目相关问题

```
1. ki query-group --scope ${scope}-memory --mode full
   → 获取项目记忆全景

2. ki query-group --scope ${scope}-memory --groups "相关Group" --mode hot,emerging
   → 查热区

3. 命中 → ki get-module-info 取原文 → 提炼回答
   未命中 → 回问用户或使用默认回答
```

### 6.2 场景：用户问个人偏好相关

```
1. ki query-group --scope user-profile --groups "相关Group" --mode hot,emerging
   → 查热区

2. 命中 → ki get-module-info 取原文 → 提炼回答
   未命中 → 使用默认行为
```

### 6.3 场景：需要结合代码知识库和项目记忆

```
1. 先查代码知识库 ${scope} → 找到模块信息
2. 再查项目记忆 ${scope}-memory → 找到相关上下文
3. 综合回答
```

## 8. 潜在风险与注意事项

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 记忆过时 | 项目技术栈变更后，旧记忆不再适用 | 定期清理机制、记忆有效期 |
| 信息冗余 | 多次沉淀相似内容 | 去重机制、相似度检测 |
| 隐私泄露 | 用户无意中暴露敏感信息 | 敏感信息过滤、权限控制 |
| 记忆污染 | 错误信息被沉淀 | 验证机制、人工确认 |
