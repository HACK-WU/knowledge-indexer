# snippet-memory 代码要点记忆

> **前置条件**：AI 已了解 MCP 工具定义和架构心智模型（ki-foundation）。
> 片段记忆存储在 `${scope}-memory` scope 下的 `通用记忆片段/` Group 中。

---

## 1. 概述

**目的**：弥补知识库偏向架构概括、缺少代码细节的问题。记录一切值得记住的代码级信息——不只是工具函数，还包括关键执行逻辑、核心流程、重要方法、通用模式等，让 AI 不需要反复阅读代码就能理解"这段代码做了什么"。

**功能**：
- 记录各类代码要点（工具函数、关键逻辑、数据模型、API 调用、配置模板、错误处理等）
- 从代码文件/文件夹批量初始化片段记忆
- 快速检索已有的代码要点，避免重复搜索代码库

**使用场景**：
- AI 编写代码前，查询是否有可复用的工具函数或已知的关键逻辑
- 用户询问"项目里有没有 XX 工具函数"、"这个功能的核心逻辑是什么"时快速响应
- 用户给出代码文件/文件夹，要求"根据这些内容构建记忆"
- AI 阅读代码时发现值得记忆的片段（工具函数、关键流程、重要模式），自动记录

---

## 2. 存储结构

在 `${scope}-memory` 下使用 ki Group 层级组织，同一分类的片段存在同一个 Relation 中。分类维度灵活，可按项目功能/领域或代码类型，也可混合：

```
# 示例 A：按功能/领域分类（优先推荐）
通用记忆片段/                   ← 顶层 Group
├── 告警/                       ← 业务功能分类
│   ├── Relation: "告警收敛"
│   ├── Relation: "通知发送"
│   └── Relation: "通用"
├── 处理套餐/
│   ├── Relation: "套餐计算"
│   └── Relation: "通用"
└── 日志查询/
    ├── Relation: "查询构建"
    └── Relation: "通用"

# 示例 B：按代码类型分类（适合通用工具）
通用记忆片段/
├── 工具函数/                   ← 代码类型分类
│   ├── Relation: "日期时间"
│   ├── Relation: "字符串处理"
│   └── Relation: "通用"
├── 关键逻辑/
│   ├── Relation: "认证流程"
│   ├── Relation: "数据处理"
│   └── Relation: "通用"
├── 数据模型/
│   ├── Relation: "实体/领域"
│   └── Relation: "通用"
├── API 调用/
│   └── Relation: "通用"
├── 配置模板/
│   └── Relation: "通用"
└── 错误处理/
    └── Relation: "通用"
```

**分类原则**：
- **优先按项目功能/领域分类**：如"告警"、"处理套餐"、"日志查询"等，与项目的业务模块对齐，这样 AI 更容易按需定位
- 大类 ≥ 2 个细分场景时才拆分，否则直接一个"通用" Relation
- 单 Relation 超过 10 个片段时考虑拆分
- **AI 根据项目实际领域自主扩展分类**，不要拘泥于固定模板

**两种分类维度可混合使用**：
- **按功能/领域**（优先）：告警、处理套餐、日志查询、用户管理... 适合业务代码记忆
- **按代码类型**：工具函数、数据模型、API调用... 适合通用可复用代码记忆
- 创建时先判断"这个片段是哪个业务功能相关的"，有明确归属则走功能分类；纯通用则走类型分类

## 3. 片段格式

片段按类型有两种格式，均保持简洁。

### 工具函数/数据模型/API/配置/错误处理 格式

```markdown
### formatDate
- **函数**: `DateUtils.formatDate(date: Date): string`
- **作用**: Date 格式化为 yyyy-MM-dd
- **路径**: `src/utils/date.ts#L42`
- **用法**: `DateUtils.formatDate(new Date())` → `"2026-06-23"`
```

### 关键逻辑/核心流程 格式

用于记录某功能的代码执行要点，说明"这段代码做了什么、怎么做"：

```markdown
### 告警收敛流程
- **入口**: `AlertConvergenceService.converge(alert)`
- **路径**: `src/alert/convergence.ts#L30-L85`
- **核心步骤**:
  1. 按策略匹配收敛规则（时间段/维度/阈值）
  2. 同维度告警合并到同一收敛组
  3. 达到阈值后触发通知，重置计数器
- **关键点**: 收敛窗口内同一维度只发一次通知；阈值按策略可配置

### JWT 认证流程
- **入口**: `AuthMiddleware.authenticate(request)`
- **路径**: `src/auth/middleware.ts#L15-L60`
- **核心步骤**:
  1. 从 Header 取 token → 验证格式
  2. 解析 payload → 校验过期时间
  3. 查缓存 → 命中直接通过；未命中查 DB 并回填缓存
  4. 将用户信息注入 request context
- **关键点**: 缓存 TTL 5 分钟；token 过期返回 401；DB 不可用时降级为仅校验签名

### 订单状态机
- **入口**: `OrderStateMachine.transition(order, event)`
- **路径**: `src/order/state-machine.ts#L50-L120`
- **状态流转**: 待支付 → (付款) → 已支付 → (发货) → 已发货 → (签收) → 已完成
- **关键点**: 已支付状态可退款回退；各状态变更触发对应事件；并发通过乐观锁控制
```

**片段命名**：`### 简短描述` — 3~8 个字，便于快速扫描。

## 4. 写入片段

### 追加写入（单个片段）

```
ki_sync_relation(
  scope: "${scope}-memory",
  group: "通用记忆片段/工具函数",
  relation: "日期时间",
  module_info: "### formatDate\n- **函数**: ...\n...",
  keywords: ["DateUtils", "日期", "格式化"]
)
```

- **追加而非覆盖**：新片段追加到已有 Relation 末尾
- **写入后刷新缓存**：`ki_query_group(scope: "${scope}-memory", mode: "full")`

### 写入前检查

写入前必须检查目标 Relation 是否已存在同名片段：
- 先 `ki_get_module_info` 获取该 Relation 当前内容
- 若已存在同名 `###` 片段 → 更新而非追加
- 若不存在 → 追加

---

## 5. 查询片段

```
① 判断需要的分类 → 如"工具函数"
② ki_query_group(scope: "${scope}-memory", groups: "通用记忆片段/工具函数", mode: "hot")
③ ki_get_module_info → 获取分类的完整片段列表
④ 匹配是否有可复用的片段
⑤ 有匹配 → 直接使用；无匹配 → 考虑是否需要新增
```

**编码前自动查询**：AI 在编写代码前，应主动查询片段记忆中是否有可复用的工具。

## 6. 初始化：从代码构建片段记忆

> **用户给出代码文件/文件夹时，AI 自动提取可复用片段并写入。**

### 触发信号

- 用户说"根据这些内容构建记忆"
- 用户说"把这个文件夹里的工具函数记录下来"
- 用户提供代码文件路径或文件夹路径，明确要求提取片段

### 初始化流程

```
① 扫描代码：读取用户指定的文件/文件夹，识别可复用的函数、类、方法
② 分类判断：优先判断"这个片段属于哪个业务功能" → 归属明确的走功能分类（如"告警"）；纯通用的工具走类型分类（如"工具函数"）
③ 提取片段：为每个可复用项生成片段格式（名称、签名、作用、路径、用法）
④ 批量写入：按分类→Relation 组织，追加写入
⑤ 刷新缓存：ki_query_group(mode: "full")
```

### 识别规则

提取片段时，重点关注以下类型：

| 代码类型 | 识别特征 | 分类 |
|----------|----------|------|
| 纯工具函数 | 无副作用、输入输出明确、通用性强 | 工具函数 |
| 核心执行逻辑 | 功能的主流程入口、关键业务步骤、状态机 | 关键逻辑 |
| 类型/接口定义 | interface、type、class（数据载体） | 数据模型 |
| HTTP 请求封装 | fetch/axios 封装、API 调用函数 | API 调用 |
| 配置对象/常量 | 配置项、枚举、常量定义 | 配置模板 |
| 错误处理机制 | try/catch 封装、错误恢复、降级策略 | 错误处理 |

**不记录**：
- 过于简单的 getter/setter
- React/Vue 组件模板（走知识库）
- 一次性的临时代码
- 纯配置值（如 `const PORT = 3000`，除非涉及多环境差异）

### 初始化示例

用户说："把 `src/utils/` 下的工具函数记录到片段记忆"

```
① 扫描 src/utils/ → 发现 date.ts、json.ts、string.ts
② 识别 extractFunction → 提取每个导出的函数
③ 分类 → 全部归入"工具函数"
④ 生成片段列表 → 按细分（日期时间/字符串处理/通用）组织
⑤ 写入对应的 Relation
```

## 7. 与知识库的边界

| | snippet-memory | codekb-skill |
|---|---------------|-------------|
| 内容 | 代码级要点（函数、关键流程、数据模型等） | 架构知识、模块职责、API 设计 |
| 粒度 | 函数/方法/核心流程级别 | 模块/系统级别 |
| 存储 | `${scope}-memory` 下 `通用记忆片段/` | `${scope}` |
| 查询 | 直接取原文（列表式） | 热区 → 原文（文章式） |

---

## 8. 禁忌

| # | 红线 |
|---|------|
| 🔴 1 | 片段内容过长（超过 5 行），应精简为 1-2 句话 |
| 🔴 2 | 写入无意义的片段（每个片段必须包含入口、路径、核心步骤/作用之一） |
| 🔴 3 | 初始化时覆盖已有 Relation（必须追加或更新同名片段） |
| 🔴 4 | 写入后不刷新缓存 |
| 🔴 5 | 不检查重复就写入（同名片段应先检查再更新） |

---

## 9. 协同

- **与 `memory-skill`**：片段记忆是项目记忆的子集，共享 `${scope}-memory` scope，Group 为 `通用记忆片段/`
- **与 `codekb-skill`**：知识库覆盖架构级，片段记忆覆盖函数级，互补不重叠
- **与 `ai-codekb-memory` 规则**：遵守 AGENTS.md 缓存规范，片段记忆的 Group 结构纳入项目记忆索引
